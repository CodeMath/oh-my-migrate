## 1. 제품 비전

### 한 줄 요약

**"FastAPI 개발자를 위한 agent-browser 스타일 DB 마이그레이션 CLI"**

AI 코딩 에이전트(Claude Code, Cursor, Codex 등)에게 **"DB 스키마를 보고 안전하게 변경하는 능력"**을 부여하는 CLI 도구.
Vercel의 agent-browser가 AI에게 웹 브라우저 조작 능력을 준 것처럼,
agent-migrate는 AI에게 데이터베이스 마이그레이션 능력을 준다.

### 핵심 철학

1. **snapshot → ref → action** 패턴: agent-browser의 설계 패턴을 DB 마이그레이션에 적용
2. **토큰 효율**: Alembic 마이그레이션 파일(50~100줄)을 10줄 이내로 요약하여 AI 에이전트의 컨텍스트 소모를 80% 절감
3. **위험도 인식**: 단순 실행이 아니라 "이 마이그레이션이 데이터를 날릴 수 있는지" 사전 분석
4. **Zero-config**: Alembic의 가장 큰 진입 장벽(env.py 설정, metadata 연결 등)을 제거
5. **Python 생태계 호환**: Alembic을 대체하는 것이 아니라, Alembic 위에 에이전트 친화적 인터페이스를 씌움

## 2. 문제 정의

### 2.1 현재 페인 포인트

#### 바이브코더 시나리오

```
1. Claude Code에게 "User 모델에 phone 필드 추가해줘"
2. Claude Code가 models.py 수정 ✅
3. DB는 아직 예전 스키마 ❌ → 앱 크래시 💥
4. "Alembic 마이그레이션 해줘" → Claude Code가 alembic 명령어 실행
5. env.py 설정 안 되어있어서 에러 → 삽질 시작
6. 결국 DB 직접 ALTER TABLE → 마이그레이션 히스토리 꼬임
7. 다음에 또 같은 문제 반복
```

#### Claude Code 하드유저 시나리오

```
1. 모델 변경 → alembic revision --autogenerate
2. 생성된 마이그레이션 파일 리뷰해야 하는데, 컨텍스트에 다 올리면 토큰 낭비
3. autogenerate가 놓치는 것들 (Enum 변경, 컬럼 rename 등)
4. 마이그레이션 적용 후 실제 DB 상태를 확인하려면 psql 별도 실행
5. 프로덕션 배포 전 마이그레이션 안전성 확인 방법 없음
```

### 2.2 근본 원인

| 문제 | 근본 원인 |
|------|-----------|
| FastAPI에 빌트인 마이그레이션 없음 | Django와 달리 "batteries-not-included" 철학 |
| Alembic 설정이 복잡 | env.py, alembic.ini, metadata 연결 등 수동 설정 필요 |
| AI 에이전트가 마이그레이션 위험도를 모름 | Alembic은 위험도 분석 기능이 없음 |
| 모델 vs DB 불일치를 실시간 감지 못 함 | Alembic autogenerate는 명시적 실행 필요 |
| 마이그레이션 출력이 AI 에이전트에 비효율적 | 사람 읽기용 포맷, 토큰 낭비 |

### 2.3 시장 기회

- FastAPI: GitHub stars 80k+, Python 웹 프레임워크 1~2위
- 바이브코딩 시장: 2026년 $4.7B, 2027년 $12.3B 전망
- AI 코딩 도구 사용자의 84%가 일상적으로 AI 코딩 도구 사용
- "Python의 Prisma Migrate" 또는 "FastAPI의 manage.py makemigrations" 포지셔닝이 빈자리

## 3. 타겟 사용자

### 3.1 Primary: 바이브코더 (FastAPI)

- Claude Code, Cursor 등으로 FastAPI 프로젝트를 개발
- Alembic 설정을 모르거나 귀찮아함
- DB 마이그레이션에서 자주 막힘
- "모델 수정하면 DB도 자동으로 바뀌었으면"

### 3.2 Secondary: Claude Code 하드유저

- FastAPI + SQLAlchemy/SQLModel로 프로덕션 서비스 운영
- Alembic을 쓰고 있지만 AI 에이전트와의 통합이 비효율적
- 마이그레이션 안전성 검증이 필요
- CI/CD 파이프라인에 마이그레이션 가드를 넣고 싶음

### 3.3 Anti-persona (타겟이 아닌 사용자)

- Django 사용자 (이미 빌트인 마이그레이션이 있음)
- Node.js/TypeScript 개발자 (Prisma, Drizzle 등이 있음)
- DBA/데이터 엔지니어 (Flyway, Liquibase 등 엔터프라이즈 도구 사용)

## 4. 제품 요구사항

### 4.1 핵심 명령어 (agent-browser 패턴)

#### `snapshot` — 현재 상태를 에이전트 친화적으로 표현

```bash
$ agent-migrate snapshot
```

**기능:**
- Python 모델 파일 자동 탐색 (SQLAlchemy, SQLModel, Pydantic 기반)
- DB 연결 정보 자동 감지 (.env, settings.py, config.py 등)
- 모델 스키마와 DB 스키마를 ref 체계로 출력
- drift(불일치) 자동 감지

**출력 포맷 (에이전트 최적화):**
```
Models:
  @m1 User      (id:int, email:str, name:str, created_at:datetime)
  @m2 Order     (id:int, user_id→@m1:int, total:decimal, status:Enum)

Database:
  @d1 users     (id:int, email:varchar, name:varchar, created_at:timestamp)
  @d2 orders    (id:int, user_id→@d1:int, total:numeric, status:varchar)

Drift:
  @m2.status: Enum(pending,paid,shipped) vs @d2.status: varchar

Migrations: 3 applied, 0 pending
```

**토큰 목표:** 모델 10개 + 테이블 10개 기준 500 토큰 이내

#### `diff` — 구체적 차이점 표시

```bash
$ agent-migrate diff
```

**기능:**
- 모델과 DB 사이의 모든 차이점을 열거
- 각 차이의 유형 분류: additive(안전), destructive(위험), ambiguous(확인 필요)

**출력 포맷:**
```
[+] @m3 Product.description: Text (model has, DB missing) — SAFE
[~] @m2 Order.status: Enum→varchar mismatch — NEEDS_MIGRATION
[!] @m1 User.username: renamed from 'name'? — AMBIGUOUS (may be drop+create)
```

#### `plan` — 마이그레이션 계획 + 위험도 분석

```bash
$ agent-migrate plan
```

**기능:**
- diff 기반으로 SQL 마이그레이션 계획 생성
- 각 단계의 위험도 분석 (SAFE / CAUTION / DANGER)
- 데이터 손실 가능성 경고
- 기존 데이터와의 호환성 체크

**출력 포맷:**
```
Plan (2 steps):

  Step 1 [SAFE]: ALTER TABLE products ADD COLUMN description TEXT;
    Impact: 0 data changes, 89 rows unaffected

  Step 2 [DANGER]: ALTER TABLE orders ALTER COLUMN status TYPE order_status
    ⚠️  Requires: CREATE TYPE order_status AS ENUM(...)
    ⚠️  23 rows with status='cancelled' — NOT in model Enum
    ⚠️  Data loss risk: 23 rows may fail conversion

Overall risk: MEDIUM
```

#### `generate` — 마이그레이션 파일 생성

```bash
$ agent-migrate generate -m "add product description"
```

**기능:**
- Alembic 호환 마이그레이션 파일 생성 (기존 Alembic 히스토리와 호환)
- 또는 raw SQL 파일 생성 (Alembic 미사용 프로젝트용)
- plan 결과의 위험도를 주석으로 포함

#### `apply` — 마이그레이션 적용

```bash
$ agent-migrate apply                # dry-run (기본)
$ agent-migrate apply --execute      # 실제 적용
$ agent-migrate apply --execute @v4  # 특정 버전만 적용
```

**기능:**
- 기본은 dry-run (실제 DB 변경 없이 결과 프리뷰)
- `--execute` 시 실제 적용 + 결과 확인
- 롤백 지원: `agent-migrate rollback @v4`

#### `guard` — CI/CD용 안전 체크

```bash
$ agent-migrate guard
```

**기능:**
- 배포 전 안전성 검증
- 모델과 DB가 동기화되었는지 확인
- 미적용 마이그레이션 존재 여부
- destructive 마이그레이션 경고
- exit code로 CI 파이프라인 제어

### 4.2 Claude Code Skill 통합

```yaml
# .claude/skills/agent-migrate/SKILL.md
---
name: agent-migrate
description: >
  DB 마이그레이션을 관리합니다. 모델 변경 시 자동으로 drift를 감지하고
  안전한 마이그레이션을 생성/적용합니다.
  "마이그레이션", "DB 스키마", "테이블 변경", "컬럼 추가/삭제",
  "migrate", "drift", "스키마 동기화" 키워드에 반응합니다.
---
```

**자동 발동 시나리오:**
1. Claude Code가 models.py 수정 → `/agent-migrate diff` 자동 실행
2. diff 결과 기반으로 마이그레이션 계획 제안
3. 사용자 승인 시 generate + apply 실행
4. 배포 명령 시 guard 자동 실행

## 5. 비기능 요구사항

### 5.1 성능

| 메트릭 | 목표 |
|--------|------|
| snapshot 실행 시간 | < 200ms (모델 20개 + 테이블 20개) |
| diff 실행 시간 | < 100ms |
| plan 실행 시간 | < 500ms (DB 데이터 검증 포함) |
| 메모리 사용량 | < 50MB |

### 5.2 안정성

- 마이그레이션 적용 실패 시 자동 롤백
- dry-run이 기본 (명시적 --execute 필요)
- destructive 마이그레이션은 이중 확인

### 5.3 호환성

- Python 3.11+
- Alembic 마이그레이션 디렉토리와 공존 (기존 히스토리 존중)
- Claude Code, Cursor, Codex, Gemini CLI 등 모든 AI 코딩 에이전트와 호환
- macOS, Linux (WSL2 포함)

### 5.4 보안

- DB 연결 정보를 로그에 출력하지 않음
- 마이그레이션 파일에 시크릿을 포함하지 않음
- `--execute` 없이는 DB에 쓰기 작업을 하지 않음

## 6. 기술 전략

### 6.1 언어 선택: 하이브리드 접근

```
┌─────────────────────────────────┐
│  CLI Entry Point (Go/Rust)      │  ← 빠른 시작, 인자 파싱
├─────────────────────────────────┤
│  Core Engine (Python)           │  ← SQLAlchemy 메타데이터, Alembic 호환
│   ├─ Model Parser (AST + Runtime)
│   ├─ DB Inspector (information_schema)
│   ├─ Diff Engine
│   ├─ Risk Analyzer
│   └─ Migration Generator
├─────────────────────────────────┤
│  Output Formatter (Go/Rust)     │  ← 에이전트 최적화 출력, ref 체계
└─────────────────────────────────┘
```

**Go/Rust의 역할:**
- CLI 바이너리 엔트리포인트 (콜드스타트 최소화)
- 출력 포매팅 (ref 체계, 토큰 카운팅)
- DB 직접 연결 (snapshot 속도 최적화)

**Python의 역할:**
- SQLAlchemy/SQLModel 모델 런타임 로딩 (AST만으로는 한계)
- Alembic 호환 마이그레이션 생성
- 기존 Python 생태계와의 통합

### 6.2 설치

```bash
# pip (Python 사용자)
pip install agent-migrate

# brew (macOS)
brew install agent-migrate

# npm (Node.js 생태계)
npm install -g agent-migrate

# cargo (Rust 생태계)
cargo install agent-migrate
```

### 6.3 Zero-config 전략

1. 프로젝트 루트에서 `*.py` 파일 스캔 → SQLAlchemy/SQLModel import 감지
2. `.env`, `settings.py`, `config.py` 등에서 DATABASE_URL 자동 추출
3. `alembic/` 디렉토리 존재 시 기존 히스토리 인식
4. 없으면 자체 마이그레이션 히스토리 관리 (`_agent_migrate_versions` 테이블)

## 7. 차별화 포지셔닝

### 7.1 기존 도구와의 비교

| | Alembic | Django Migrations | Prisma Migrate | **agent-migrate** |
|---|---|---|---|---|
| 대상 프레임워크 | SQLAlchemy | Django | Prisma | FastAPI/SQLAlchemy/SQLModel |
| 초기 설정 | 복잡 | 자동 | 자동 | **Zero-config** |
| AI 에이전트 최적화 | ❌ | ❌ | ❌ | **✅ (snapshot+ref)** |
| 위험도 분석 | ❌ | ❌ | Preview | **✅ (SAFE/CAUTION/DANGER)** |
| 토큰 효율 | 사람용 출력 | 사람용 출력 | 사람용 출력 | **에이전트 최적화** |
| CI/CD 가드 | 별도 구성 | 별도 구성 | 별도 구성 | **빌트인 (guard)** |
| dry-run 기본 | ❌ | ❌ | ✅ | **✅** |

### 7.2 핵심 메시지

- **바이브코더에게:** "모델 바꾸면 DB도 따라온다. 마이그레이션을 몰라도 된다."
- **시니어에게:** "Alembic 위에 위험도 분석과 AI 에이전트 통합을 얹었다."
- **에이전트 생태계에게:** "agent-browser가 웹을 줬듯이, 우리는 DB 마이그레이션을 준다."

## 8. 로드맵

### Phase 1: MVP (4주)

**목표:** "모델 바꾸면 agent-migrate가 알아서 해준다" 기본 플로우

- `snapshot`: SQLAlchemy 2.0 모델 파싱 + PostgreSQL 스키마 조회
- `diff`: 모델 vs DB 차이 감지 (컬럼 추가/삭제/타입변경)
- `plan`: SQL 마이그레이션 계획 + 기본 위험도 (SAFE/DANGER)
- `generate`: Alembic 호환 마이그레이션 파일 생성
- `apply`: dry-run + --execute
- Claude Code SKILL.md 제공
- PyPI 배포

### Phase 2: 안정화 (4주)

- SQLModel 지원
- Enum, JSON, Array 등 복잡한 타입 diff
- FK / relationship 변경 감지
- 데이터 마이그레이션 필요성 감지 + 경고
- `guard` 명령어 (CI/CD)
- `rollback` 명령어
- MySQL / SQLite 지원

### Phase 3: 에이전트 생태계 (4주)

- 다중 AI 에이전트 지원 (Cursor, Codex, Gemini CLI)
- skills.sh 등록
- Go/Rust 바이너리 엔트리포인트 (콜드스타트 최적화)
- brew, npm, cargo 배포
- `agent-migrate watch` (파일 변경 감시 → 자동 diff)

### Phase 4: 확장 (이후)

- Tortoise ORM, Django ORM 지원
- 멀티 데이터베이스 환경
- 마이그레이션 히스토리 시각화
- 팀 협업 기능 (마이그레이션 충돌 감지)
